Anthropic · Claude Code
The string match heard around the dev world

Claude Code is quietly refusing — or upcharging — commits that mention "OpenClaw."

twitter.com/theo · via HN 865 points 497 comments

Anthropic appears to have shipped a server-side filter that, by simple regex, downgrades or rejects requests from Claude Code sessions when the model sees the words "OpenClaw" or "Hermes" in a commit message, file, or diff. The community read is unambiguous: Anthropic is throttling open-source coding harnesses that wrap their API and resell the capacity, especially under flat-rate plans where heavy power users blow through the unit economics.

What set HN on fire wasn't the policy — most agreed unlimited plans on a frontier model were never going to last — it was the implementation. A plain string match, with no whitelist for Claude Code's own system prompt or tool surface, means legitimate developers working on adjacent open-source projects, or just discussing the harness in a README, are getting silently degraded. Anthropic eats its own dog food on Claude Code; this looks like ops triage, not policy.

For an orchestrator like you, the read-through is sharper: the price of using Claude through someone else's wrapper is now structurally different from using it directly. Bring-your-own-key for OSS harnesses is becoming the safe default. The era of "unlimited Claude for $20" is officially in the rearview.

Didn't they think — we need to make sure Claude Code is never banned?

Anthropic · Privacy

Opus 4.7 can name strangers from four sentences they wrote.

theargumentmag.com · Kelsey Piper 115 points 58 comments

Kelsey Piper paste-tested Opus 4.7 with snippets from her own unpublished writing — incognito, no memory, no account context, also verified through the raw API. It correctly identified her, again and again. Other HN commenters reproduced the result with their own drafts on the first try. One reader fed a 475-word blog post and the model nailed it in one shot.

The capability isn't memory; it's stylometry at frontier scale. The model has internalized enough of the public web that the way you write — sentence rhythm, vocabulary cadence, argumentative tics — is now a fingerprint. Earlier Claudes couldn't do this reliably. Opus 4.7 can. The Argument piece reads it for what it is: the end of casual pseudonymity on the open web for anyone with even a modest writing footprint.

The mitigations are bleak. Run your writing through a local model to neutralize tone, and you've destroyed the voice that made it worth reading. Don't, and frontier models will increasingly link your aliases together with one prompt. Expect "stylometric anonymizer" to become a category before the year is out.

I can never talk to an AI anonymously again. Neither can you.

AI Supply Chain · Security

Shai-Hulud burrowed into PyTorch Lightning — and dumped 2,200 sets of stolen creds on GitHub.

semgrep.dev 299 points 97 comments

Versions 2.6.2 and 2.6.3 of pytorch-lightning on PyPI were published with malicious payload code by an attacker who had compromised the maintainer's PyPI credentials. The GitHub repo was clean; the poison was injected post-build, directly to the index. Anyone who ran pip install lightning in the affected window had tokens, env vars, and cloud secrets exfiltrated.

The signature move is the Shai-Hulud one: the malware then publishes the stolen credentials into thousands of attacker-created GitHub repos, naming them after Dune characters and lightly obfuscating the secrets with double base64. The convention abuses the fact that everyone — security researchers, secret scanners, attackers — is already polling GitHub for leaked tokens. Exfiltration via the global commons. A repo search returned 2,200+ matching repos created in the last 24 hours.

If you're on the AI side: pin your training-stack deps, audit anything that ran pip install -U this week, and rotate any cloud creds that were resident in a dev machine that touched Lightning. The official fix is to stay on 2.6.1 until 2.6.4 ships. The deeper signal: as AI tooling becomes the most active package surface on the internet, it is also the juiciest supply-chain target.

Bless the Maker — and pin your dependencies.

Browsers · LLM APIs

Mozilla draws a hard line on Chrome's plan to expose a built-in LLM to every web page.

github.com/mozilla 564 points 207 comments

Google has been quietly shipping the Prompt API in Chrome — a JavaScript surface that lets any web page call an on-device Gemini Nano model. Mozilla's standards-positions team has now formally objected. Their argument: a web standard that exposes a non-deterministic, vendor-controlled language model to every script in the browser isn't a web feature, it's a vendor moat dressed in standards clothing.

The technical critique is real. Different browsers shipping different models means web apps will quietly behave differently — sometimes catastrophically so — depending on the user's browser. Outputs aren't testable, security boundaries against prompt injection are undefined, and there's no interop story when the underlying model changes weekly. Mozilla's position is a useful brake on a feature that, in Chrome's framing, was being treated as inevitable.

The strategic subtext is more interesting: Chrome wants in-browser AI to be a first-class web platform primitive, owned by browsers. Mozilla, with no frontier model and a tiny share, is arguing that MCP-style external connections (under user control) are the right substrate. Both can be right — but only one is going to win the default position in the next major Chrome release.

A standards body should not enshrine the output of a single company's frontier lab.

Agentic AI · DX

A full coding agent — Anthropic + OpenAI, seven tools, REPL — in 400 lines of shell.

pu.dev · Show HN 75 points 21 comments

pu.sh is one POSIX shell file. sh + curl + awk, nothing else. It implements an interactive coding-agent loop with seven tools (bash, read, write, edit, grep, find, ls), supports both Anthropic's and OpenAI's tool-use surfaces, carries reasoning items across turns, does auto-compaction and checkpoint/resume, and ships 90 no-API tests. The author confesses up front: most of the awk was written by Claude and Codex; he can't read large parts of his own code.

That admission is the actual story. A year ago, "build a Claude Code clone in one shell file" was a joke. Now it's a weekend project — and the result is dependency-free, vendor-agnostic, and small enough to audit in an afternoon. The community pushback (the minification makes it unreadable; that's vibe-coded spaghetti) is fair, but it misses the macro point: the floor for "good enough to replace Claude Code as a scaffold" has cratered.

For anyone running automation pipelines on top of frontier APIs, this is the new ambient baseline. The agent harness is no longer a moat — the model and the eval loop are. Expect to see this pattern repeat across every "agent CLI" startup in the next 12 months.

The agent loop is tiny. Everything else is DX and hardening.

Developer Tools

Honker stuffs queues, pub/sub, streams and a cron into one SQLite file.

honker.dev 158 points 49 comments

The pitch is local-first heresy: when your app needs background jobs and a message bus, you don't reach for Redis and Celery — you point everything at a single SQLite file. Honker provides durable queues, fan-out streams, pub/sub, and a cron scheduler, all using SQLite's PRAGMA data_version as the wake signal. Workers poll the version counter every millisecond, a roughly 3µs read; when it ticks, they know something committed somewhere and they wake up to scan the queue.

HN's reception was sharp. tptacek pointed out the giveaway clue — busy-polling a SELECT every millisecond does not, in fact, sound better than a kernel file watcher, regardless of how the marketing copy frames it. Several commenters noted that for the kind of small-to-medium app that picks SQLite as a substrate, a couple of percent of CPU spent on a polling thread per worker is probably fine. Others muttered that Oban already supports SQLite and is battle-tested.

The interesting principle isn't this library specifically — it's the larger trend of treating SQLite as a serious production substrate, not just an embedded toy. Litestream, Litestack, Turso, and now Honker are all variations on the same bet: "Postgres + Redis + a queue" is overkill for 90% of apps that ship.

No Redis, no broker — just the file on disk you already have.

Privacy · Web Tracking

LinkedIn is silently fingerprinting your browser for 6,278 extensions.

6,278
Extensions Probed Per Page Load
404privacy.com 299 points 121 comments

404privacy.com pulled apart LinkedIn's page scripts and found the company is checking, on every load, whether 6,278 specific browser extensions are installed — by attempting to fetch their web-accessible resources. The result vector is encrypted and embedded into every subsequent request. The list isn't security-focused; it's a wholesale fingerprint and behavioral classifier.

This is the kind of tracking that's been theoretically possible for a decade and rarely deployed at this scale. The encrypted payload is the giveaway — they're not just using it for one feature; they're funneling it into the broader analytics pipeline, and the encryption is there to make it harder for third parties (or regulators) to audit. The technique would never survive a German or French DPA filing if it were named honestly.

For anyone working on AI products, the implication is unflattering: as AI agents become the new browser-side actors, this kind of side-channel will be one of the first defenses platforms reach for. Expect "agent vs. site" fingerprinting wars to ramp up in 2026.

If you can detect it, you can sell it.

Energy · AI Infrastructure

Belgium reverses course — the nuclear plants stay on.

dpa-international.com 712 points 656 comments

Belgium has formally suspended its nuclear decommissioning program. Plants that were scheduled to be wound down through the late 2020s will instead be kept running — and in several cases relicensed for further life-extension. The proximate driver is the same one tipping every European energy decision in 2026: industrial baseload demand is climbing faster than renewables can pencil, and the marginal customer is now AI data centers.

It's worth pausing on the shift. Belgium's nuclear phase-out was, until 2023, the European green movement's flagship policy commitment. The fact that it has now been reversed — under center-left coalition cover — tells you exactly how much the political math has changed. Bringing Anthropic and OpenAI to Europe means promising them grid capacity, and there is no path to that capacity by 2027 without nuclear.

For the broader AI story: every major US hyperscaler now has direct power-purchase agreements with nuclear operators (Microsoft / Three Mile Island, Amazon / Talen, Google / Kairos). Europe's version of this is now visible in policy. The next AI bottleneck isn't compute — it's electrons.

No nuclear, no data centers. No data centers, no European AI.

Surveillance · History

The book excerpt: how a quiet AT&T technician walked into the EFF and uncovered Room 641A.

mitpress.mit.edu 379 points 108 comments

MIT Press's The Reader ran a long excerpt from a forthcoming book on Mark Klein, the AT&T technician who in 2006 told the Electronic Frontier Foundation about a secret room at 611 Folsom Street in San Francisco where the NSA was splitting every fiber-optic line passing through the facility. The piece is part history, part memoir — a careful look at what it costs to blow that particular whistle.

Two decades on, Room 641A reads differently. The "is the NSA reading all email?" debate has been replaced by "is your model provider training on your prompts?" — and the answer is, in many cases, yes by default unless you explicitly opt out. The technical machinery is the same: a fiber splitter, a couple of beam-splitting prisms, an analyst on the other end. What's changed is who the analyst is, and that the analyst is now a model.

For an industry that has spent five years arguing about model safety and three months arguing about agent permissions, this is the kind of grounding read that gets the proportions right. The privacy story doesn't start in 2022. It started in a switching closet in San Francisco — and the people who told us about it paid for it.

Algorithms · Developer Culture

Daniel Lemire: you can, in fact, beat the binary search.

lemire.me · Daniel Lemire 226 points 110 comments

Lemire's blog post takes the textbook claim — binary search is asymptotically optimal for comparison search on sorted arrays — and demolishes the practical version of it. Through branchless implementations, cache-aware layouts (think Eytzinger order), and a careful read of modern CPU branch predictors, his variants run several times faster than the canonical loop, even though they perform the same number of comparisons in the worst case.

It's a small reminder that "asymptotically optimal" and "fast on a 2026 CPU" have been different categories for at least a decade. Memory hierarchy, branch prediction, and SIMD have done more for real-world search performance than any genuinely new algorithm in years. The interesting work is in lining your data up so the chip can see it; the binary search itself is almost incidental.

The HN thread is a delightful nerd-snipe: at least a dozen commenters posted alternative implementations with benchmarks, and the consensus is that nobody should ever type out a binary search in 2026 — you should pull in std::lower_bound or whatever your runtime ships, and trust it to be branchless. Reach for handwritten loops only when you've measured a real bottleneck.

Asymptotically optimal is not what the cache thinks.
